# LLMPageTrans 二期规划

## 1. 背景

当前一期版本已经具备以下核心能力：

- 面向 Chrome 扩展形态，支持对当前网页进行 LLM 翻译。
- 通过克隆页面 DOM 到 `iframe` 并回写译文，尽量保留原页面布局。
- 支持双栏对照 / 单栏译文两种展示模式。
- 支持 Markdown 词库、超大请求分批、页面批次并发、基础滚动同步。
- 支持用户直接配置 API URL、API Key、Model Name 与目标语言。

从现有实现看，一期已经验证了“网页结构保留式翻译”这条路线是可行的，但也暴露出几个明显瓶颈：

- 缺少可重复的离线回归测试素材与验证机制，迭代复杂页面时容易退化。
- 翻译前无法预估 token 消耗与成本，用户使用门槛较高。
- 模型接入仍是“单 URL + 单模型”配置方式，不利于扩展多供应商。
- 对复杂页面、动态页面、长页面的结构保真与体验仍有提升空间。
- 尚未支持 PDF 文档翻译，能力边界主要停留在普通 HTML 页面。

因此，二期目标不是零散补功能，而是将产品从“可用 Demo”推进到“可持续迭代的实用工具”。

## 2. 二期目标

二期聚焦五个主题方向：

1. 建立离线测试与回归机制，降低复杂页面迭代风险。
2. 提供翻译前 token / 成本预估，增强用户决策能力。
3. 抽象统一的模型供应商接入层，支持多家模型平台。
4. 提升复杂网页的翻译质量、结构保真与展示体验。
5. 增加 PDF 翻译能力，扩展到论文、报告、技术文档场景。

## 3. 设计原则

- 渐进增强：优先在现有架构上演进，不推倒重来。
- 能力分层：把“文档解析、内容抽取、模型调用、结果渲染”解耦。
- 可测试：新能力优先设计回归样例与验收标准。
- 可配置：复杂能力默认给出合理默认值，同时允许高级用户调整。
- 可降级：复杂页面与 PDF 场景允许按能力梯度回退，避免彻底失败。

## 4. 范围定义

### 4.1 二期纳入范围

- 离线测试样例、测试入口、回归清单。
- token 用量和成本预估。
- 多模型供应商接入抽象与配置 UI。
- 复杂页面翻译增强。
- PDF 翻译 MVP。

### 4.2 二期暂不纳入范围

- 云端账号体系、多人协作、历史翻译云同步。
- 全站批量抓取翻译。
- 完整的文档编辑器能力。
- 面向移动端浏览器或 Safari / Firefox 的适配。
- 高保真 PDF 双语导出编辑器级排版能力。

## 5. 现状与问题拆解

### 5.1 测试层

当前项目几乎以人工在线页面验证为主，问题包括：

- 线上页面内容随时变化，不适合作为稳定回归样本。
- 某次修复滚动同步，可能意外破坏复杂布局页面。
- 无法系统比较“译文是否覆盖完整”“布局是否明显错位”“性能是否退化”。

### 5.2 成本感知层

当前点击翻译即直接请求模型，用户无法提前知道：

- 页面大致会消耗多少 token。
- 不同模型下的成本差异。
- 超大页面是否值得翻译、是否需要切换低成本模型。

### 5.3 模型接入层

当前模型调用逻辑主要写在 `background.js` 中，和具体请求格式耦合较深，存在问题：

- 不同平台的 Base URL、鉴权方式、模型列表、参数字段可能不同。
- UI 上只有通用文本输入，不适合长期扩展。
- 后续如果接入文本模型、视觉模型、推理模型，会进一步增加条件分支复杂度。

### 5.4 页面处理层

当前方案本质上是“抽取文本节点 -> 批量翻译 -> 原位回写”。这在普通静态页面上效果不错，但对以下情况不够稳定：

- 强动态渲染页面。
- 内部滚动容器较多的页面。
- 结构复杂的技术文档、表格、代码块、公式混排页面。
- 懒加载、折叠块、虚拟列表、长目录导航页面。

### 5.5 PDF 场景

PDF 与普通网页差异较大，至少存在三种输入形态：

- 浏览器内嵌 PDF Viewer 页面。
- 真实 PDF 文件 URL。
- 扫描版 PDF 或文本层缺失的 PDF。

如果直接沿用 HTML 文本节点方案，无法覆盖 PDF 的内容解析、页级结构重建和多模态兜底。

## 6. 二期功能规划

### 6.1 模块 A：离线测试与回归体系

#### 目标

建立一套可重复执行、可持续扩展的测试样本库和人工回归流程，为后续复杂页面与 PDF 能力提供基础设施。

#### 方案

- 新增 `test_pages/` 或 `fixtures/` 目录，存放离线 HTML 样例。
- 首批样例建议覆盖：
  - 普通文章页
  - 技术博客页
  - 含代码块与表格的文档页
  - 含懒加载图片 / 复杂 CSS 的页面
  - 含内部滚动容器的页面
  - 近似论文页面
- 支持人工提供样例页面，统一沉淀为固定测试集。
- 增加一个“本地测试页加载说明”文档，约定如何在浏览器中打开离线样例并进行验证。
- 输出回归 checklist，至少包含：
  - 是否成功翻译
  - 是否存在大面积漏翻
  - 是否出现明显布局错位
  - 分屏 / 单栏切换是否正常
  - 滚动同步是否可接受
  - 翻译耗时是否异常

#### 建议扩展

- 后续增加一个“调试模式”，把抽取到的文本节点数、批次数、预计 token、实际耗时打印出来，便于测试记录。

#### 验收标准

- 至少沉淀 8 到 12 个稳定离线测试页。
- 每次较大改动后，可以按统一流程完成一次回归。
- 对每个样例形成简短备注，说明该样例主要覆盖哪类风险。

### 6.2 模块 B：Token 与成本预估

#### 目标

在用户真正发起翻译前，预估本次请求规模与成本，帮助其决定是否继续、是否切换模型、是否启用分批。

#### 方案

- 在内容抽取完成后、真正调用模型前，先生成待翻译 `textMap`。
- 基于模型类型做 token 估算：
  - 基础版：字符数 / 单词数启发式估算。
  - 进阶版：引入与目标模型兼容的 tokenizer 进行更准确估算。
- 预估结果至少展示：
  - 待翻译文本节点数
  - 预估输入 token
  - 预估输出 token
  - 预估总 token
  - 预估成本区间
  - 是否超过建议阈值
- UI 上建议增加“先预估再翻译”或“翻译前展示预估弹层”。
- 对于没有配置单价的供应商，允许只展示 token 规模，不展示成本金额。

#### 实现建议

- 新增独立的 `estimation` 模块，不把估算逻辑直接混写进 `background.js`。
- 供应商配置中增加模型单价字段：
  - input price / 1M tokens
  - output price / 1M tokens
- 对分批模式同步展示：
  - 预计分成多少批
  - 每批大概 token 规模

#### 验收标准

- 用户点击翻译前可看到预估结果。
- 对同一页面，不同模型的估算结果可对比。
- 估算误差控制在“足以辅助决策”的范围内，先不追求财务级精度。

### 6.3 模块 C：多模型供应商接入

#### 目标

把当前“单接口配置”升级为“供应商抽象 + 模型配置管理”，至少支持：

- 火山方舟
- OpenRouter
- ModelHub
- DeepSeek

#### 方案

- 抽象统一接口，例如：
  - `buildRequest(provider, model, messages, options)`
  - `sendChatCompletion(providerConfig, payload)`
  - `normalizeResponse(provider, response)`
- 区分供应商元数据与用户配置：
  - 供应商类型
  - Base URL
  - API Key
  - 模型列表
  - 是否支持多模态
  - 价格信息
- UI 调整建议：
  - 先选择供应商
  - 再选择或填写模型
  - 仅展示该供应商需要的配置项
- 配置存储建议从“零散字段”升级为“结构化 provider config”。

#### 参考策略

- 可以参考 `chatbox` 的 provider 设计思路，但不建议完整照搬。
- 本项目更适合做“轻量 provider adapter”，避免把桌面聊天应用的复杂配置全部引入。

#### 风险点

- 不同供应商接口看似兼容 OpenAI 格式，实际字段常有差异。
- 多模态模型与纯文本模型的参数并不完全一致。
- 流式 / 非流式、推理参数、上下文限制需要统一包装。

#### 验收标准

- 新增任意一个供应商时，不需要修改核心翻译流程。
- 用户可以在 UI 中切换供应商并保存独立配置。
- 至少 4 家供应商完成基本文本翻译链路验证。

### 6.4 模块 D：复杂页面翻译增强

#### 目标

让产品从“适合普通网页”提升到“对技术内容、复杂布局页面也有较高可用性”。

#### 方案拆分

1. 内容抽取增强
   - 区分正文、导航、页脚、广告、按钮文案、代码块。
   - 优先翻译高价值正文区域，降低噪音内容占比。
   - 针对代码、公式、pre/code、表格单元格做更细粒度规则。

2. 页面分块策略
   - 以 DOM block 为单位分组，而不是仅按文本节点固定批次切分。
   - 保留块级上下文，减少前后句被切散导致的术语漂移。
   - 对标题、段落、列表、表格使用不同处理策略。

3. 展示效果优化
   - 对过长译文引发的布局膨胀做样式兜底。
   - 对图片说明、表格、注释区单独处理，减少错位。
   - 在译文页面加入“原文 / 译文对照高亮”调试能力，便于定位异常块。

4. 交互稳定性增强
   - 优化复杂滚动容器识别。
   - 对懒加载内容提供“重新抓取并增量翻译”的补救路径。
   - 对 SPA 页面切换增加重新注入或重新翻译提示。

#### 技术建议

- 在抽取阶段引入“节点分类”和“块级 ID”，让后续估算、翻译、调试、回写都围绕块进行，而不是围绕零散文本节点进行。
- 为复杂页面增加调试开关，输出：
  - 跳过了哪些节点
  - 哪些区域被识别为正文
  - 每个块的文本长度与翻译状态

#### 验收标准

- 技术博客、文档页、含表格页面的整体表现明显优于一期。
- 复杂页面上的漏翻、误翻、错位问题可定位、可复现、可回归。
- 用户对“为什么这个区域没翻译”有基本可解释性。

### 6.5 模块 E：PDF 翻译 MVP

#### 目标

优先支持“可解析文本层的 PDF”，为技术论文、白皮书、说明书等场景提供最小可用能力。

#### 推荐路线

二期建议采用“两段式方案”，不要一开始就全面依赖截图 + 多模态模型。

第一阶段：文本层 PDF

- 使用 `pdf.js` 或浏览器可用的 PDF 解析能力提取每页文本项、位置和页码。
- 以“页”为基本单位做翻译和渲染。
- 输出形态优先选择：
  - PDF 专用阅读视图
  - 左原文页 / 右译文页，或上下切换视图
- 先追求“可读”和“结构基本对应”，不强求逐像素复刻原 PDF 排版。

第二阶段：扫描版 / 复杂版 PDF

- 对无文本层 PDF，引入 OCR 或截图 + 多模态识别作为兜底路径。
- 可以页级截图发送给多模态模型，但成本高、速度慢、稳定性弱，应放在二期后半段或标记为实验能力。

#### 不建议的做法

- 直接把整份 PDF 截图后交给多模态模型整体翻译。

原因：

- token 和成本极高。
- 页面多时响应时间不可控。
- 译文与页内结构对齐困难。
- 对公式、双栏论文、脚注、图表引用的稳定性较差。

#### MVP 范围

- 支持文本层 PDF 的页级解析与翻译。
- 支持基础页码导航。
- 支持段落级或块级译文展示。
- 对扫描版 PDF 明确提示“当前仅部分支持”。

#### 验收标准

- 至少能稳定处理一批技术论文或说明书 PDF。
- 用户可以浏览原文与译文，并按页定位内容。
- 对不支持的 PDF 类型给出明确降级提示，而不是静默失败。

## 7. 架构演进建议

为了支撑二期，建议将现有实现从“单文件堆逻辑”演进为更清晰的模块结构。

### 7.1 推荐模块划分

- `providers/`
  - 各模型供应商适配器
- `translation/`
  - prompt 构建、批处理、结果合并
- `estimation/`
  - token 估算、成本估算
- `page-analysis/`
  - DOM 抽取、节点分类、正文识别、分块
- `pdf/`
  - PDF 解析、页级渲染、OCR / 多模态兜底
- `fixtures/`
  - 离线测试页面与说明

### 7.2 数据流建议

统一抽象为：

1. 输入源识别：HTML 页面 / PDF
2. 内容抽取：节点或块
3. 规模预估：token / 成本 / 分批信息
4. 供应商调用：文本模型或多模态模型
5. 结果回写：网页 `iframe` 或 PDF 专用视图
6. 调试记录：耗时、批次、错误、降级路径

## 8. 里程碑建议

### M1：基础设施与可观测性

目标周期：1 周

- 沉淀离线测试样例与回归说明
- 增加调试日志和基本指标输出
- 完成二期代码目录重构的最小骨架

### M2：成本预估与 Provider 抽象

目标周期：1 到 2 周

- 完成 token / 成本预估 MVP
- 完成 provider adapter 抽象
- 接入 2 家供应商验证链路

### M3：多供应商与复杂页面增强

目标周期：2 周

- 扩展到 4 家供应商
- 完成内容抽取增强、块级翻译策略
- 优化复杂页面展示与滚动同步

### M4：PDF MVP

目标周期：2 周

- 接入 PDF 文本层解析
- 实现 PDF 专用翻译视图
- 完成若干论文 / 文档 PDF 验证

## 9. 优先级建议

如果资源有限，建议按照以下顺序实施：

1. 离线测试体系
2. Provider 抽象
3. Token / 成本预估
4. 复杂页面增强
5. PDF MVP

原因如下：

- 没有测试体系，后续复杂功能迭代容易反复回退。
- 没有 provider 抽象，多模型接入会迅速变成技术债。
- 成本预估用户价值高，且实现投入相对可控。
- 复杂页面增强是主体验提升项，但依赖前面的可测试性与可观测性。
- PDF 最具扩展价值，但解析链路和展示链路都更复杂，适合作为二期后段重点。

## 10. 风险与对策

### 10.1 供应商接口不一致

对策：统一 adapter 接口，对每家供应商做独立请求与响应归一化。

### 10.2 复杂页面效果难以稳定

对策：引入离线页面样例、块级调试能力和降级策略，不追求一次性覆盖所有站点。

### 10.3 成本估算误差较大

对策：先做启发式估算，界面上明确写明“预估值”，后续再替换为更精确 tokenizer。

### 10.4 PDF 扫描版成本过高

对策：二期优先支持文本层 PDF；扫描版只做实验特性或明确提示暂不支持。

## 11. 建议产出物

二期完成后，建议形成以下产出：

- 结构化的 provider 接入代码
- 一套可复用的离线测试样例
- 翻译前 token / 成本预估 UI
- 复杂页面增强版翻译链路
- PDF MVP
- 一份回归 checklist 与问题定位说明文档

## 12. 结论

二期的核心不是单独补上“五个需求点”，而是补齐三个产品基础能力：

- 可测试
- 可扩展
- 可解释

只要这三个基础打牢，后续无论接入更多模型、支持更复杂网页，还是扩展到 PDF、多模态和更多文档类型，都会顺畅很多。建议以“测试体系 + provider 抽象 + 成本预估”作为二期前半程主线，以“复杂页面增强 + PDF MVP”作为二期后半程主线推进。
